# RevLoop — Case Study

**AI-powered growth analytics for early-stage products.**

Live demo: https://rev-loop.vercel.app
Source: https://github.com/ryanshaon/RevLoop

---

## The problem

Early-stage teams collect plenty of data — signups, page views, campaign spend — but most of it sits scattered across analytics tools, spreadsheets, and dashboards that show *what happened* without saying *what to do about it*. A founder can see "signups are up 12% this week" and still have no idea whether those signups are worth anything, whether the activation flow is leaking users, or which of their five marketing channels is quietly burning money.

The gap isn't data. It's a decision layer on top of the data — something that takes funnels, retention, churn signals, and channel economics and turns them into a short list of "here's what's broken, here's what's working, here's what to try next."

RevLoop is my attempt at building that layer, end to end, as a real shippable product rather than a notebook full of charts.

---

## Target user

Primary: an early-stage founder or growth lead with a product that has users but no dedicated analytics team — someone who needs to make a call on Monday morning about where to spend the next $500 of marketing budget or which onboarding step to fix first.

Secondary: a product manager who inherited a dashboard nobody trusts and wants metrics that are actually wired to real definitions (a "retained" user means something specific, not just "logged in once").

I built and seeded the dataset around a fictional product, **Campus Connect Demo** — a college events app — so every number on the dashboard traces back to a believable, internally consistent story instead of generic placeholder data.

---

## Product decisions

A few choices that shaped the whole build, in the order they mattered:

**Decision layer over chart library.** Every page exists to answer a specific question someone would actually ask — "where are we losing people," "who's about to churn," "what should we test next" — rather than offering a generic metrics explorer. The Insights and Experiments pages exist specifically to close the loop from observation to action.

**Calendar-aligned retention, not signup-relative.** Early versions measured retention "days since signup" per user, which makes cohorts impossible to compare honestly. Switched to calendar week cohorts so "week 2 retention" means the same calendar window for every user in a cohort — the only way the retention heatmap is actually trustworthy.

**Rule-based insights engine, not an LLM call.** Every risk, opportunity, and recommended experiment on the Insights page is generated by deterministic logic against real query results — not a prompt to an LLM. That means every sentence is traceable to a number you can verify yourself, there's no API cost or latency, and it's honest about being a v1: a future iteration could add an LLM summarization layer on top of this same deterministic core, but the core itself needed to be trustworthy first.

**Soft delete on experiments, not hard delete.** Cancelling an experiment sets its status rather than removing the row — growth experiments are historical record, and "we tried this and it failed" is exactly as valuable as "we tried this and it worked."

## Architecture

```
Browser
  │
  ▼
Next.js frontend (Vercel)
  │  HTTPS / JSON
  ▼
FastAPI backend (Render)
  │
  ├──▶ PostgreSQL (Supabase) — users, events, campaigns, experiments
  │
  └──▶ scikit-learn churn model — in-process, no network hop
```

Frontend and backend are deployed and scaled independently. The ML model loads into the FastAPI process at startup rather than living behind its own service — at this scale, a network call to a separate model server would add latency for no benefit, and the model artifacts (under 6KB) are small enough to ship with the backend.

---

## Data model

Seven tables, designed around a real event-tracking schema rather than a denormalized "users with a churn column":

- **organizations** — the tenant (one row: Campus Connect Demo)
- **users** — signup date, acquisition channel, plan type
- **events** — the actual behavioral log: every `page_viewed`, `signup_completed`, `event_registered`, etc., timestamped per user
- **campaigns** — spend per acquisition channel, for CAC/ROI math
- **revenue_events** — payment records, separate from generic events
- **experiments** — hypothesis, target metric, status, result summary
- **insights** — generated risk/opportunity/experiment records

Funnel, retention, and churn are all computed *from the events table* rather than stored as precomputed fields — meaning the same raw log can answer new questions later without a schema migration.

Seed data: 800 users, 20,000+ events, 6 acquisition channels, 8 weeks of activity, deliberately uneven across channels (Referral converts well, Paid Ads converts poorly) so the insights and channel-comparison logic has something real to find.

---

## ML churn model

**The label:** a user is "churned" if they've had no meaningful activity (page views, registrations, returns) for 14+ days, measured against the latest event date in the dataset rather than the real calendar date — since this is simulated historical data, not a live stream.

**The features:** 14 engineered signals per user — days since signup, days since last active, session count, event count, channel, plan type, and short-window activity counts (last 7/14 days).

**The model decision worth explaining:** my first version used Random Forest and hit 100% accuracy. That's not a win — it's a red flag. `days_since_last_active` is both an input feature *and* the literal basis of the churn label, so the model had simply learned to repeat the label definition back to me. It produced binary 0/1 confidence scores, which are useless for ranking who to email first.

I switched to Logistic Regression, which gives 253 distinct, genuinely calibrated probability scores (Brier score 0.011, log loss 0.047) instead of overconfident binary outputs — far more useful when the real question is "who are the top 50 highest-risk users this week," not "is this one user churned, yes or no."

Training and serving share one `features.py` module so the exact same feature logic runs at train time and inference time — a common, easy-to-miss source of silent model drift in real systems.

---

## Insights engine

Takes the outputs of the funnel, retention, channel, and churn logic and runs them through prioritized rule sets:

- **Risks** ranked by severity: churn rate, retention health, biggest funnel leak, weakest channel
- **Opportunities** ranked by impact: best-performing channel, underused retention levers, upsell signals
- **Recommended experiments** ranked by expected value: which funnel step to A/B test first, which channel to reallocate budget toward

The output reads like a short weekly memo a human analyst might write — specific numbers, specific channel names, specific next actions — rather than a generic "engagement is down" alert.

---

## Experiment tracker

A lightweight CRUD board: create a hypothesis, set a target metric, move it through planned → running → completed/cancelled, log the result. This is the piece that turns RevLoop from a read-only dashboard into something that closes the loop — an insight on Monday becomes a tracked experiment, and the result feeds back into whether the next insight should be trusted.

---

## Deployment

- **Frontend:** Vercel, auto-deploys from `main`
- **Backend:** Render free tier (Python web service) — note: sleeps after inactivity, first request can take 30–50 seconds to wake
- **Database:** Supabase (PostgreSQL, Mumbai region)

All three are free-tier services wired together with environment variables — no infrastructure cost to keep this live as a portfolio piece.

---

## Limitations

Being upfront about what this is and isn't:

- **Simulated data, not live users.** Every number is real arithmetic on a generated dataset, not actual customer behavior — strong models on synthetic data don't guarantee strong models on messy real-world data.
- **No authentication.** Single-tenant by design; multi-tenant access control was explicitly out of scope to keep the build focused.
- **Free-tier cold starts.** The backend sleeping after inactivity is a real (if minor) UX cost of running this for free.
- **Insights engine is rule-based, not generative.** It's deterministic and explainable, which I think is the right v1 choice — but an LLM summarization layer is a natural next iteration once the underlying logic is trusted.

---

## What I learned

The most valuable lesson wasn't a tool or a framework — it was learning to be suspicious of a model that performs *too* well. A 100% accuracy churn model felt like success until I worked out why it was happening, and fixing that taught me more about feature/label leakage than any tutorial would have.

The second lesson was about scope. I had a longer list of features I wanted to build — LLM-generated summaries, multi-tenant support, real SDK-based event tracking — and cutting all of it to ship a focused, working v1 was harder than adding more would have been.

---

## Business value

RevLoop demonstrates the same thinking a Product Analyst, Growth Analyst, or early PM hire needs day one: turning a pile of metrics into a prioritized, defensible "do this next." It's not just full-stack engineering — it's the product judgment of deciding which five questions actually matter, the data discipline of defining "retained" and "churned" precisely enough to trust, and the honesty to flag a model's weakness instead of hiding behind a good-looking accuracy number.
